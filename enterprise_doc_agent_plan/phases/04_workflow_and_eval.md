# 第四阶段：工作流编排、评测与异步任务

## 阶段目标

把系统从“能跑”升级成“更像业务系统”。

这一阶段重点不是增加花哨功能，而是补齐企业落地能力：

- 工作流编排
- 任务队列
- 日志与监控
- 结果评测
- 人工校对入口

## 你需要学习的知识

- LangGraph 基础
- 状态机思维
- Redis 基础
- Celery 或 RQ
- 任务状态追踪
- 结构化输出校验
- 评测指标设计

## 这一步为什么重要

企业环境下，文档处理可能很慢，也可能失败。

所以你需要解决下面问题：

- 上传后不能一直等接口阻塞
- 失败后怎么重试
- 怎么看每次处理用了多少时间
- 怎么知道哪一步出错
- 怎么让人工补救

## 建议新增模块

```text
app/
  agents/document_agent.py
  workflows/document_graph.py
  workers/task_worker.py
  services/eval_service.py
```

## 推荐的工作流节点

你可以先设计下面这些节点：

1. 文档接收
2. 文档类型判断
3. 文本/表格/OCR 路由
4. 翻译或摘要
5. 知识入库
6. 结果重建
7. 评测与日志记录

## 你最该补的业务能力

- 异步任务处理
- 失败重试
- 人工审核状态
- 质量评分
- Token 成本记录

## 推荐的评测指标

- 文档处理耗时
- 表格还原成功率
- OCR 识别准确率
- 术语命中率
- 问答命中率
- 单任务 Token 成本

## 最小实现建议

这一阶段不要求你一次全做完。

建议优先顺序：

1. 先做异步任务
2. 再做日志和状态追踪
3. 最后再补简单评测

## 这一阶段完成后的项目定位

完成后你就可以把项目包装为：

- 企业智能文档处理 Agent 平台

这时你不仅有模型调用，还有：

- 工作流
- 任务状态
- 可观测性
- 可评测性

## 这一阶段适合写进简历的话

- 基于 LangGraph 与异步任务队列重构文档处理工作流，实现多阶段路由、状态追踪、失败重试与质量评测，提升系统稳定性与可运营性。
