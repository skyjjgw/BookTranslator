# 第三阶段：RAG 知识库与文档问答

## 阶段目标

让文档处理结果不只是“导出翻译文件”，还可以：

- 文档入库
- 语义检索
- 基于文档问答
- 自动生成摘要和关键词

这一步完成后，项目的业务价值会明显上升。

## 你需要学习的知识

- RAG 基础原理
- 文本切分
- Embedding 向量化
- 向量数据库基础
- 检索 + 生成流程
- 提示词中注入上下文

## 推荐学习栈

- `LangChain`
- `Chroma` 或 `FAISS`
- `sentence-transformers` 或模型平台 Embedding

## 这一步的业务意义

企业最痛的并不只是翻译，而是：

- 资料太多，找不到
- 老文档无法复用
- 新人不熟悉历史材料

所以你要把项目升级成：

- 上传文档后自动入库
- 后续支持提问和检索

## 建议新增模块

```text
app/
  services/chunk_service.py
  services/vector_store_service.py
  services/qa_service.py
```

## 处理流程建议

1. 文档解析后拿到文本块
2. 对文本块切分
3. 生成向量并写入向量库
4. 用户提问时检索相关片段
5. 把检索结果拼进提示词，再调用大模型生成答案

## 你需要新增的接口

- `POST /documents/upload`
- `POST /documents/index`
- `POST /documents/ask`
- `GET /documents/{id}/summary`

## 这一步新增代码的重点

- 文本切块
- 向量入库
- 检索召回
- 带上下文问答

## 最小实现建议

只做下面这条最小链路即可：

- 上传 PDF
- 解析文本
- 切块入库
- 支持一个提问接口

不要一上来就做：

- 多轮记忆
- 权限控制
- 混合检索

## 这一阶段完成后的项目定位

你就可以把项目描述成：

- 面向企业知识管理的文档理解与问答系统

## 这一阶段适合写进简历的话

- 构建基于向量检索的文档知识库链路，支持解析结果入库、语义召回与上下文增强问答，提升历史文档复用效率。
